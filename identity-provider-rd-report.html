Identity Provider & Authentication Architecture

Claims A · Provider Matrix B · Shortlist C · Architecture D · Onboarding E · Approval Workflow F · Auth Flow G · SSO vs API H · Migration Plan

The current TCS-admin-manual model does not scale. Customer-admin-led onboarding is achievable with any of the shortlisted providers today. The identity/authorization boundary is clean and enforceable.

✅ Claim 1 — Burden

Every evaluated provider has a first-class invitation API. Manual TCS admin loop is removable on day one of any shortlisted provider.

✅ Claim 2 — Boundary

IdP JWT carries only sub, email, email_verified. Platform backend owns all tenant, role, and entitlement context.

✅ Claim 3 — Self-Service

All three shortlisted providers support customer-admin roles. Invitation, MFA, and user management are API-driven — TCS never in the routine path.

The Architecture Principle That Matters Most

Identity Provider — Credentials & Login Only
sub (user ID) email email_verified mfa_enrolled account_status tenant_id role subscription entitlements
↓ JWT: sub · email · email_verified · exp (thin by design)
Platform Backend — Authorization & Business Context
tenant_id platform_user_id idp_sub (FK) role entitlements subscription_tier document_access

Swapping IdP providers = update one JWKS validation URL + re-implement AuthService adapter. Zero changes to business logic.

Auto-Approval Rules

✓ Domain whitelist

Email domain in org's approved domain list → auto-approve + assign default role

⚠ High-privilege role

Invited role is "Admin" or "Finance" → require manual approval regardless of domain

✓ Role threshold

Invited role is "Viewer" or "Read-only" → auto-approve (low blast radius)

Email domain not in org's approved list → require manual approval

Invitation created by org admin directly → auto-approve (admin already vouched)

⚠ Default (no rule matches)

No configured rule applies → require manual approval (fail safe)

SSO vs API Integration Analysis

A clear distinction applies to every downstream system. These are different integration patterns — most confusion comes from conflating them.

User SSO

The same human logs into multiple apps with one set of credentials. Federated login via SAML/OIDC. The IdP handles the federation.

Backend API Integration

System-to-system data sync. Platform ↔ external service using OAuth2 service account or long-lived token. No human in the loop. IdP not involved.

Deliverable H

Migration Plan

Current State

  • Users created manually by TCS admins — no self-service
  • No IdP in production — assuming no existing Auth0/Cognito/similar
  • User data in platform DB; password storage format unknown (see Open Questions)
  • Active sessions: assumed minimal given current manual flow
Phase 0

Parallel Setup

  • Provision IdP tenant
  • Implement AuthService adapter
  • Stand up JWKS validation middleware (off by default)
  • Set up webhook endpoint /webhooks/idp-events
  • Add nullable idp_sub column to platform_users
  • Build customer admin UI (invite, manage, approval queue)
  • Implement approval workflow engine
Phase 1

New Users Only

  • Enable new onboarding for new tenants only
  • Customer admins use invite flow
  • Monitor: invite delivery, signup completion, MFA enrollment rates
  • JWT validation active for new users; legacy auth still works for existing
  • Zero risk to live users
Phase 2

Existing User Migration

  • Export existing user emails (no password hashes)
  • Bulk-create users in IdP via Management API (status: password-reset-required)
  • IdP sends "set up your account" email to each user
  • User sets new password + enrolls MFA
  • Platform links idp_sub on first IdP login
  • Old auth credential disabled after 30-day grace period
Phase 3

Legacy Sunset

  • Confirm all active users have idp_sub populated
  • Disable legacy auth middleware
  • Deactivate MIGRATION_PENDING accounts after 90 days with notification
  • Remove old credential storage
  • TCS user-creation process officially retired
