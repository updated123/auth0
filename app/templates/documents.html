How Resource Access Works
1
User logs in via Auth0

Auth0 issues a thin JWT containing only sub, email, and email_verified. No document permissions in the token.

2
Platform DB maps sub → entitlements

The backend validates the JWT, extracts sub, and looks up the user's org_id, role, and entitlements[] from the platform database.

3
Entitlements gate document visibility

Each document has a required entitlement. The backend checks required_entitlement in user_entitlements before returning the resource. Auth0 is never consulted.

Backend Access Check Logic
# Platform backend — resource-level access check # Called on every document request after JWT validation def can_access(user_entitlements: list[str], required_entitlement: str) -> bool: """ Check if user has the required entitlement. Wildcard '*' grants access to everything (admin). """ if "*" in user_entitlements: return True return required_entitlement in user_entitlements # Example: resolving user context from JWT sub async def get_document(doc_id: str, jwt_sub: str, db: AsyncSession): user = await db.get_user_by_sub(jwt_sub) # Platform DB lookup doc = await db.get_document(doc_id) if not can_access(user.entitlements, doc.required_entitlement): raise HTTPException(status_code=403, detail="Entitlement required") return doc # Auth0 was never consulted
Auth0 never knows about documents. It issues a thin JWT with only sub and email. The platform backend maps sub → entitlements → document access at request time. Changing a user's entitlements takes effect immediately — no token re-issue needed.
